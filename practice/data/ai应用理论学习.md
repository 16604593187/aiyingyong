# AI 应用开发理论学习笔记

## 第一层：大模型 API 调用

大模型 API 调用的核心是维护一个消息列表，每条消息包含 role 和 content 两个字段。role 有三种：system 定义助手的角色和行为规则，权重高于 user；user 是用户的输入；assistant 是模型的回复。

模型本身是无状态的，它不记得任何上下文。所谓"记住上文"，本质是每次调用时把历史消息列表完整发给模型，模型才能理解上下文。这意味着对话历史越长，每次调用消耗的 token 越多，费用越高，速度越慢。

Context Window 是模型单次能处理的最大 token 数。超出后必须截断历史，否则 API 报错。截断策略有两种：直接丢弃最早的消息（简单但会丢失信息），或者对早期对话做摘要压缩（保留信息但需要额外调用）。

截断时机很重要：应该在 assistant 回复追加完之后再截断，而不是在 user 消息追加后立即截断。否则 history 长度为奇数时，截断点会落在 assistant 消息上，导致实际保留数比预期多一条。

System prompt 的权重高于 user 消息，但不是绝对的。如果 system prompt 本身允许被覆盖，模型会服从 user 的指令。

## 第二层：向量嵌入（Embedding）

Embedding 是把文本映射到高维向量空间的技术。语义相近的文本，在向量空间中的方向也相近。这不是巧合，而是模型被大量语义相似的句子对训练的结果。

衡量两个向量相似度最常用的是余弦相似度，计算的是两个向量夹角的余弦值，结果在 -1 到 1 之间，1 表示完全相同，0 表示无关。

为什么用余弦相似度而不用欧氏距离？因为欧氏距离受向量长度影响，长文本生成的向量模更大，会导致"长文本之间天然更相似"的假象。余弦相似度只看方向，归一化掉了长度的影响，更适合语义比较。

Embedding 的重要局限：它捕捉的是话题相关性，而不是语义正确性。否定词、转折词对向量距离的影响极小。"这个方法适合初学者"和"这个方法不适合初学者"的相似度高达 0.77，差距只有 0.08。这意味着 RAG 检索时可能把语义相反的内容也召回来。

numpy 的向量化机制把循环下沉到 C 语言层面执行，比 Python 循环快几十倍。广播机制允许不同形状的数组直接运算，numpy 自动处理维度对齐。

## 第三层：向量数据库

向量数据库解决的核心问题是：知识库有几千几万个片段时，如果每次查询都和所有片段逐一计算相似度，复杂度是 O(n)，文档多了会很慢。向量数据库内部用 HNSW 等索引结构，把查询复杂度降到接近 O(log n)。

Chroma 是一个轻量的本地向量数据库，适合原型开发。使用 PersistentClient 可以把数据持久化到本地磁盘。创建 collection 时指定 hnsw:space 为 cosine，表示使用余弦距离作为相似度度量。

Chroma 的 query 接口返回的 distances 字段，在 cosine 空间下等于 1 减去余弦相似度。所以 score = 1.0 - distance 才是真正的相似度分数。这个转换公式只在 hnsw:space=cosine 时成立，使用其他距离度量时公式不同。

用文本内容的 SHA256 哈希作为文档 ID，配合 upsert 而不是 add，可以实现幂等导入：同一段文本无论导入多少次，知识库里只保留一份，不会重复。

Chroma 的 metadata 字段只接受 str、int、float、bool 四种类型。其他类型需要先序列化成 JSON 字符串再存入，否则运行时报错。

懒加载是一种优化模式：模块被 import 时不立即初始化重型资源（如模型、数据库连接），而是在第一次实际使用时才初始化。这样可以避免 import 时的冷启动延迟。

## 第四层：RAG（检索增强生成）

RAG 的完整流水线：对用户问题做 Embedding，在向量库里检索 Top-K 相关片段，把检索结果和用户问题拼成 Prompt，调模型生成答案，返回答案和参考来源。

为什么要切片而不是把整篇文档塞进去？原因有两个：一是 context window 限制，长文档可能超限；二是检索精度，整篇文档的向量是所有内容的语义平均，会变得模糊，切片后每个片段语义更集中，检索更准。

固定长度切片的问题：按固定字符数截断会把完整的语义单元切成两半，导致片段向量语义不完整，检索相似度偏低。递归字符切片按优先级依次尝试分隔符（段落、换行、句号），在保证片段不超过最大长度的同时尽量保留语义完整性，效果更好。

RAG 的 Prompt 组织方式：检索结果通过 system prompt 或专门的 context 块注入，明确告诉模型"只基于以下参考资料回答，如果资料中没有相关内容就说不知道"。这样可以减少模型幻觉，让回答有据可查。

知识库内容质量直接决定 RAG 效果。代码块、目录树、符号等噪音内容会拉低检索质量。规划类文档（描述"要做什么"）不如知识性文档（解释"是什么、为什么"）适合作为 RAG 的知识源。

不是所有问题都需要走 RAG。闲聊、简单问候等不需要知识支撑的问题，走 RAG 反而会产生奇怪的回答。工程上用 Router 模式解决：用一个小模型做意图分类，判断是否需要检索，再路由到不同的处理链。

本地 Embedding 模型（如 bge-small-zh）在 CPU 上的推理耗时和文本长度正相关，短文本约 10-50ms，正常长度文本约 100-400ms。程序启动时预热模型可以避免第一次查询时的冷启动延迟。

语义缓存（Semantic Cache）是 NLP 场景下有效的缓存方案：对新问题计算 Embedding，和缓存中已有问题做相似度比较，相似度超过阈值则直接返回缓存答案。普通字符串匹配缓存在自然语言场景下命中率极低，因为同义不同形的表达太多。

## 第五层：记忆与持久化

持久化记忆的必要性：为了保证ai助手能在多次开启对话后依然能记住自己做过什么、用户问了什么，持久化记忆是必要的。

为什么选择JSON存储：JSON 是人类可读的纯文本，出错时可以直接打开检查；不需要额外依赖；结构天然匹配 messages 列表。

四种记忆策略（滑动窗口、懒压缩、分级存储、向量化记忆）各自的适用场景和局限：滑动窗口逻辑简单，没有内容压缩，适合短期多轮且前后对话内容长期相关度较低的对话，但是对话长时完全没有截断前对话记忆。压缩本身有成本，每次压缩都是一次 API 调用，如果每轮都压缩，成本反而比不压缩更高。懒压缩通过只在历史超过阈值时压缩来控制压缩频率，但是会出现摘要的摘要这种信息丢失加重的问题。分级存储同时保留了近期原文的完整性和远期历史的压缩摘要，是在 token 成本和信息保留之间取得平衡的方案，但是不适合非线性记忆与跨对话记忆。向量化记忆适合非线性跨会话记忆，但它的代价是每轮对话都要写入向量库，查询时也要做检索，比线性历史复杂得多；而且检索出来的是碎片化片段，不保留对话的顺序和连贯性。

prompt 结构中各部分的顺序及其背后的注意力机制依据（Lost in the Middle）：注意力机制中首尾高中部低，除了注意力强度的考虑，还有语义绑定的原因——检索片段是专门为回答当前问题而存在的，两者紧邻放置，模型更容易把参考资料和问题关联起来，所以对prompt中各部分顺序处理为系统规则+对话历史+检索片段+用户问题。

memory和RAG边界：当前会话用线性历史、跨会话长期记忆用向量检索。对话有顺序依赖，"你刚才说的 X"这类引用需要完整连续的上下文，向量检索返回的是碎片，会破坏连贯性。跨会话历史量大，不能全量塞进 prompt，向量检索可以按需取用相关片段。

加入RAG后的prompt结构：系统规则+对话历史+检索片段+用户问题。

## 第六层：router和语义缓存

router的三种实现方式及选择依据：三种实现方式分别为关键词分类、LLM意图分类、基于向量化后的余弦相似度分类。自然语言的语义不能单纯从关键词考虑，这会导致其关键词库极其臃肿且难以维护。而向量化后余弦相似度分类是基于语义相关性而非语义正确性来判断的，否定词转折词对其影响极小，并不能确定其语义的含义。LLM 理解语义，分类效果更好，代价是多一次 API 调用；但 Router 只传当前用户请求、不带历史，token 消耗极低，可以用更小的模型进一步压缩成本，整体可接受。

router的成本优化：1.需要router分类的传入参数可以仅限于当前用户的请求。2.router可以采用参数比较低的模型，用于降低成本。

语义缓存的原理和阈值选择逻辑：语义缓存是根据用户当前请求的向量化结果与缓存库中的答案进行比对，如果相似度大于阈值，则返回缓存结果，跳过LLM调用。阈值选择第一需要判断助手的用途，如果是闲聊，则其阈值较低，为0.85-0.90，如果是知识助手，阈值需要提高到0.90-0.95，如果是医疗或法律咨询，则需要提高到0.95-0.98确保不受语义歧义影响，第二需要根据用户意图判断进行分级阈值设立：如果是闲聊，则降低阈值，如果是知识库检索请求，则提高阈值。如果是上文检索，则跳过缓存检索，直接进行后续步骤。

为什么按返回标签分级阈值：助手的设计目的可能固定，但是用户请求并不固定。一个知识助手，用户的请求完全可能是闲聊或知识检索，需要根据他们的意图不同选择合适的阈值。

followup不缓存的原因：followup涉及到上下文的内容，相同的语义如果上下文不同那他们的表达意图也就完全不同，语义缓存完全不能解决这类问题，需要模型自行回答。

router+cache组合后的成本结构变化：缓存命中：Router 调用→ 直接返回，0 次主 LLM 调用。缓存未命中 + chitchat/followup：Router 调用 + 主 LLM 调用，无 RAG 检索成本。缓存未命中 + rag：Router 调用 + Embedding 向量化 + 主 LLM 调用。和没有 Router/Cache 的原始版本相比，只有"缓存未命中 + rag"这条路径成本持平，其他路径都有不同程度的降低，高频重复问题趋近于零成本。

## 第七层：Agent 工具调用

### Agent 的本质

Agent 和普通 LLM 调用的根本区别在于：普通调用是"用户问 → 模型答"的单次闭环，Agent 是"用户请求 → 模型决策 → 工具执行 → 结果反馈 → 模型再决策"的多步循环。模型不再只是回答者，而是具备了主动调用外部能力的决策者。

### 四种经典 Agent 范式

ReAct（Reasoning + Acting）：最经典的范式。模型每步输出 Thought（我要做什么）、Action（调用哪个工具和参数）、Observation（工具返回结果），循环直到得出最终答案。推理过程透明可追踪，适合多步推理场景，但每轮都需要 LLM 调用，成本较高。

Function Calling：不靠 Prompt 约定格式，而是把工具定义成结构化 JSON Schema 传给模型，模型直接输出结构化的工具调用指令，由外部代码执行后把结果塞回对话继续。结构化输出稳定，不需要手动解析，依赖模型原生支持（DeepSeek 支持）。工具边界清晰的场景首选。

Plan-and-Execute：分两阶段，先由 Planner 生成完整执行计划，再由 Executor 按计划逐步执行。适合任务结构固定、步骤可预先规划的长任务。缺点是计划生成后若中途出现意外，灵活性差，可能需要重新规划。

Self-Consistency / Reflection：模型生成答案后，由 Critic 角色审查质量，不满意则反馈重新生成，循环直到满足质量标准。输出质量高，但调用次数多，成本高，可能陷入无限循环。适合对输出质量要求极高的场景。

### Function Calling 的工作机制

工具以 JSON Schema 形式定义，包含工具名、描述、参数类型和必填项。调用流程分四步：第一步，把工具定义列表和用户消息一起发给模型；第二步，模型判断是否需要调用工具，如果需要则在响应中返回 tool_calls 字段，包含工具名和参数，此时模型不直接生成答案；第三步，外部代码解析 tool_calls，实际执行对应函数，得到结果；第四步，把工具执行结果以 role: tool 的消息追加回对话，再次调用模型，模型根据工具结果生成最终答案。

模型可能在一次响应中请求调用多个工具（parallel tool calls），也可能在得到工具结果后再请求调用下一个工具（sequential tool calls），整个过程由模型自主决策，外部代码只负责执行和传递结果。

### Function Calling 与 RAG 的边界

RAG 是"检索后注入上下文"，检索动作由代码控制，模型不知道也不参与检索决策。Function Calling 是"模型主动请求工具"，模型自己判断何时需要搜索、搜什么。两者可以共存：把 RAG 检索封装成一个工具，让模型在需要时主动调用，比固定每次都走 RAG 更灵活，模型可以决定不检索（闲聊场景）或多次检索（问题复杂时）。

### 工具调用的消息结构变化

引入 Function Calling 后，消息列表中会出现两种新的消息类型：role: assistant 且包含 tool_calls 字段的消息（模型请求调用工具），以及 role: tool 且包含 tool_call_id 和执行结果的消息（工具返回结果）。这两种消息必须成对出现且顺序正确，否则 API 报错。记忆持久化时需要完整保存这些消息，否则恢复会话后模型无法理解上下文。

### 工具描述（description）的工程意义

工具描述是模型决策的唯一依据。模型看不到函数代码，只看 JSON Schema 里的 description 字段。描述写得不准确，模型就会在不该调用的时候调用、在该调用的时候不调用，或者传错参数。比如 search_web 的描述需要明确说明"涉及最新事件、时效性数据、知识库中没有的内容时使用"，否则模型可能对所有问题都触发搜索，或者从不搜索。参数的 description 同理，"建议简洁精准"这类提示能引导模型生成更高质量的参数值。工具描述的质量直接决定 Agent 的行为质量，是 Function Calling 中最容易被忽视但影响最大的工程细节。

### tool_choice 参数的控制粒度

tool_choice 参数控制模型是否以及如何调用工具，有三种模式：none 表示禁止调用任何工具，模型只能文字回复；auto 表示由模型自主判断是否需要调用（最常用，模型根据问题决策）；指定具体工具名表示强制调用该工具（适合需要保证工具一定被调用的场景）。实际开发中 auto 是默认选择，但在测试阶段临时切换为 none 可以快速验证模型文字回复路径是否正常，切换为指定工具可以单独测试某个工具的参数生成质量。

### 历史截断的安全边界

前几层的截断逻辑假设消息以 user + assistant 成对出现，直接按条数截断是安全的。引入 Function Calling 后，一轮用户请求可能对应多条消息：user、assistant（tool_calls）、tool（结果1）、tool（结果2）、assistant（最终回复）。如果截断点落在 tool_calls 消息和 tool 消息之间，messages 里会有 tool_calls 但没有对应的 tool 结果，API 会返回格式校验错误。安全的截断策略是只在 role=user 处截断：收集所有 user 消息的下标作为合法截断点，从倒数第 N 个截断点处切片，保证每一段历史都从 user 开始、tool_calls/tool 组始终完整。这个规则和前几层保证截断在 user 处的直觉一致，只是实现方式从"按偶数条数截断"变成了"按 user 下标截断"。

### 与前几层的整合思路

Layer 7 的 Agent 实际上覆盖并替代了 Layer 6 Router 的部分职责：Router 通过意图分类决定"要不要检索"，Function Calling 通过工具描述让模型自己决定"要不要调用搜索工具"，不需要额外的意图分类步骤。搜索结果由模型主动拉取，而非 RAG 流水线强制注入，模型对信息的使用更灵活。实际系统中两种方案不是非此即彼的关系：对于结构固定、必须检索的问题，RAG 注入更可靠；对于需要模型动态判断检索时机的复杂任务，Function Calling 更合适。知识库检索本身也可以封装成工具，让模型决定什么时候调向量库、什么时候联网搜索，把 RAG 和 Web Search 统一纳入工具体系管理。
